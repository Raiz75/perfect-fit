# Laravel Conventions — Security Practices

**Laravel version reference:** 12.x / 13.x

---

## 1. XSS Prevention in Blade — `{{ }}` vs `{!! !!}`

```blade
{{-- Correct — auto-escaped output, safe by default --}}
<p>{{ $comment->body }}</p>
```

```blade
{{-- Wrong — raw output of user-controlled content --}}
<p>{!! $comment->body !!}</p>
```

```blade
{{-- Correct use of {!! !!} — only for content sanitized/generated server-side, never raw user input --}}
<div>{!! Str::markdown($post->trusted_markdown_field) !!}</div>
{{-- where $post->trusted_markdown_field is authored by trusted staff, or run through a sanitizer like HTMLPurifier --}}
```

**Rule:** Default to `{{ }}` for all output — it escapes HTML entities and is the only safe default for anything derived from user input. `{!! !!}` is reserved for content that is either generated by the server (not user-supplied) or has been explicitly passed through an HTML sanitizer. Never use `{!! !!}` directly on a user-submitted field (comment body, bio, profile name) — that's a direct stored-XSS vector.

## 2. SQL Injection Avoidance — Parameter Binding vs Raw Queries

```php
// Correct — Eloquent/Query Builder parameter-binds automatically
Order::where('status', $request->input('status'))->get();
DB::select('select * from orders where status = ?', [$request->input('status')]);
```

```php
// Wrong — string interpolation into raw SQL
DB::select("select * from orders where status = '{$request->input('status')}'"); // SQL injection
DB::raw("select * from orders where user_id = " . $userId); // same problem
```

**Rule:** Eloquent and the Query Builder parameter-bind by default — use them normally and injection isn't a concern. If a raw query is genuinely necessary (complex reporting SQL, DB-specific functions), always use `?` placeholders or named bindings and pass values as a separate array — never interpolate a variable directly into a SQL string, including inside `DB::raw()`. `orderBy`/column names can't be parameter-bound, so if a column name comes from user input, validate it against an explicit allow-list before use.

## 3. Mass Assignment — Layered Defense (Authorization + Validation + `$fillable`)

```php
// Correct — three independent layers, each catching a different failure mode
class UpdateOrderRequest extends FormRequest
{
    public function authorize(): bool
    {
        return $this->user()->can('update', $this->route('order')); // layer 1: is this user allowed to update *this* order?
    }

    public function rules(): array
    {
        return ['status' => ['required', Rule::enum(OrderStatus::class)]]; // layer 2: is the input itself valid?
    }
}

class Order extends Model
{
    protected $fillable = ['status']; // layer 3: even with valid, authorized input, only these columns are writable
}

$order->update($request->validated());
```

```php
// Wrong — relying on $fillable alone as the only defense
class Order extends Model
{
    protected $fillable = ['status', 'total', 'user_id']; // now any authorized caller can rewrite total/user_id too
}

$order->update($request->all()); // no validation, no explicit field allow-list at the call site
```

**Rule:** Treat authorization, validation, and `$fillable` as three independent layers, not substitutes for one another:
- **Authorization** (Policy/Form Request `authorize()`): can *this* user act on *this* record at all.
- **Validation**: is the submitted data well-formed and within acceptable values.
- **`$fillable`**: even given a valid, authorized request, which columns can mass-assignment ever touch.

Never call `update($request->all())` — always pass `$request->validated()` (or an explicit array) so validation is actually enforced, and keep `$fillable` scoped to only the columns that specific update path should ever change, even if the model has other mass-assignable columns used elsewhere.

---

## Quick Reference

| Concern | Convention |
|---|---|
| Blade output | `{{ }}` by default, `{!! !!}` only for sanitized/non-user content |
| SQL | Parameter binding always; never interpolate into raw SQL/`DB::raw()` |
| Column names from user input | Validate against an explicit allow-list |
| Mass assignment | Layer authorization + validation + `$fillable`, use `validated()` not `all()` |
